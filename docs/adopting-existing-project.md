# 在存量项目中渐进引入

目标不是把历史整理得完美，而是建立一个清晰的切换点，让新工作从此只有一个事实来源。

## 第一阶段：只读盘点

运行 Skill 的审计，不写文件。确认：

- 当前任务记录在哪里；
- 哪些任务仍然有效；
- 是否已经使用 Issue、PR、Actions 或其他看板；
- 默认分支和发布方式；
- 可重复执行的 lint、test、build 命令；
- 是否存在长期分支、未合并工作或共享数据库。

保留历史记录。不要为了“看起来整齐”删除旧任务文档或重写 Git 历史。

## 第二阶段：建立切换点

记录一个明确日期和规则，例如：

> 从 2026-08-01 起，所有新增工作以及仍需继续执行的旧任务都使用 GitHub Issues；旧的 tasks.md 仅作只读历史。

把仍在进行的任务迁移成 Issue。每个 Issue 保留旧记录链接或原始编号。不要迁移已经完成且没有审计价值的所有历史条目。

## 第三阶段：创建最小 Project

先使用少量状态：

`Backlog → Ready → In progress → Blocked / In review → Done`

只添加实际用于排序的字段，例如 Priority、Area、Size、Risk。不要一开始复制复杂 Scrum 工具中的所有字段。

## 第四阶段：用真实任务试点

选择 3–10 个可以独立交付的任务：

1. 补齐成果、范围、验收标准和验证方式；
2. 标出 sub-issues 和依赖；
3. 每个任务使用独立分支；
4. 通过 PR 交付并关联 Issue；
5. 记录流程中缺少的信息，而不是立即增加大量模板字段。

## 第五阶段：先稳定 CI，再强制 CI

把已经能在本地执行的检查放进 GitHub Actions。让它在几个 PR 上运行，修复偶发失败、超时、缺失依赖和测试数据隔离问题。

只有当 CI 稳定并且检查名称已经出现在 GitHub 中，才把它设为默认分支的 required status check。否则可能把团队锁在无法合并的状态。

## 第六阶段：停止双写

旧文档只保留：

- 历史记录；
- GitHub Project 链接；
- 新流程的切换日期；
- 无法迁移的背景信息。

不要继续手工同步状态。需要离线或低延迟访问时，使用可重新生成的只读快照。

## 第七阶段：再启用托管

至少让几个真实 PR 完整跑通评审、CI 和分支规则，再创建一个持久主管 Automation。刚开始保留 `per_turn` 合并政策，验证人工升级边界和连续失败处理后，再考虑只对低风险 PR 开放 `qualified_auto_merge`。

存量项目不应在迁移第一天就启用可写的事件驱动 Agent Action，也不应同时运行多个覆盖同一范围的定时主管。

## 回滚

如果试点不合适：

- 暂停新增强制规则；
- 保留已创建的 Issue 和 PR 作为历史；
- 移除或放宽 branch protection，而不是重写提交历史；
- 根据试点反馈简化字段和模板。
