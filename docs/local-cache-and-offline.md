# GitHub 本地缓存与离线使用

## 什么在本地

Git 保存的是代码对象和引用。执行 `git fetch` 后，本地会保存远端分支当时的提交位置，但不会下载 Issue 评论、PR 审批或 Project 字段。

`.github/` 是普通的版本控制目录，常见内容包括：

- Issue 表单模板；
- PR 模板；
- Actions workflow；
- CODEOWNERS；
- Dependabot 和其他仓库配置。

它不是 GitHub 数据库的离线镜像。

## 快照策略

`snapshot_github.py` 使用 GitHub CLI 查询常用 Issue 和 PR 元数据，原子写入：

`.codex/cache/github-snapshot.json`

快照包含来源仓库、生成时间和查询结果。建议：

- 产品盘点、只读总结：缓存未超过 15–60 分钟时直接读取；
- 准备修改 Issue/PR：先刷新；
- 合并、关闭、分配负责人、判断 CI：始终读取远端最新状态；
- 离线工作：明确说明所依据快照的时间。

缓存目录应该被 Git 忽略。它可以删除和重建，绝不能成为可编辑的任务事实来源。

## 为什么不提交快照

提交快照会造成：

- 每次远端状态变化都产生无意义代码提交；
- 多个 agent 产生缓存合并冲突；
- 评论、账号或内部链接意外进入公开仓库；
- 读者无法判断代码中的快照是否仍然新鲜。

如果确实需要版本化报告，应生成明确标注时间范围的发布报告，而不是伪装成实时任务数据库。
